title: "Chrome Extension Web Store Publishing"
description: "Publish an MV3 extension with offscreen + host permissions without getting blocked in Chrome Web Store review."
---

Use this checklist when submitting a new extension build (or updating an existing one) to the Chrome Web Store.

## Pre-submit checklist

1. Bump `manifest.json` version.
2. Build your release ZIP with all runtime files included.
3. Verify release permissions are minimal (remove localhost/dev-only hosts).
4. Confirm your privacy policy URL is reachable publicly.
5. Align store listing text with actual extension behavior.

## Common publish blockers

If Chrome Web Store shows:

- "A justification for host permission use is required"
- "A justification for offscreen is required"

Go to **Privacy practices** and fill the missing fields.

## Copy-paste text for Privacy practices

Use these as templates and replace placeholders with your app-specific details.

### Single purpose description

```text
[Extension name] helps users [primary user outcome]. Users can [top action 1], [top action 2], and [top action 3]. Any background processing is limited to functionality required to deliver this purpose.
```

### `storage` justification

```text
The extension needs the "storage" permission to save [specific user settings/data] so functionality persists across browser sessions. Data is stored in Chrome extension storage (`chrome.storage.local` or `chrome.storage.sync`) and used only to support the extension's stated purpose.
```

### `offscreen` justification

```text
The extension uses an offscreen document to run background logic that requires a window/DOM context not available in a Manifest V3 service worker. The offscreen document is used only for internal processing required by extension functionality and does not present user-facing UI.
```

### Host permission justification

```text
The extension requests host permissions for [host patterns from manifest] so it can communicate with required backend APIs/services over HTTPS/WebSocket to provide core functionality. These permissions are not used to inject scripts into websites or read page content.
```

### Remote code justification (if form requires text)

Select: **No, I am not using Remote code**

```text
All JavaScript and WebAssembly used by this extension are packaged locally within the extension bundle. No remote scripts, remote modules, eval of downloaded code, or externally hosted WASM are executed.
```

## Example (ZeroGPU-powered extension)

If your extension uses ZeroGPU Browser SDK in an offscreen document, use:

- Host patterns: `https://*.zerogpu.ai/*`, `https://*.workers.dev/*`
- Offscreen explanation: SDK runtime requires `window`/DOM context not available in MV3 service worker
- Remote code status: "No" if all JS/WASM are bundled locally in package

## Data usage recommendations

- If your extension does not collect/transmit user data to your servers, leave user-data categories unchecked.
- Certify all three required Chrome Web Store disclosures.
- Ensure your public privacy policy wording matches your submission form.

## Submission flow

1. Upload new package.
2. Open **Privacy practices** and complete all required fields.
3. Save draft.
4. Run final review of listing, screenshots, and privacy URL.
5. Submit for review.

## Packaging sanity check

For MV3 offscreen architecture, ensure release ZIP includes:

- `manifest.json`
- `background.js`
- `offscreen.html`
- `offscreen.js`
- `vendor/zerogpu-browser-sdk.umd.js`
- `vendor/transformers.min.js`
- popup files and icon assets

Missing offscreen/vendor files is a common reason for post-approval runtime failures.
